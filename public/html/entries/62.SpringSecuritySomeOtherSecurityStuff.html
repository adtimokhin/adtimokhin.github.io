Spring Security. Some other security stuff

What will we cover?

I want to talk about 4 other security features that we might want to use in our applications:

    • Logout of users
    • Handling "forbidden" error
    • Encrypting passwords
    • CSRF protection

Logout

Logout is something that Spring Security handles for us. When we logout the authentication token will no longer be valid.

Spring allows us to configure the logout option inside our SecurityConfig file.

This is how it is done:

SecurityConfig.class:


@Configuration

@EnableWebSecurity

@ComponentScan ("com.adtimokhin.security")

public class SecurityConfig extends WebSecurityConfigurerAdapter {


    @Override

    protected void configure (HttpSecurity http) throws Exception {

        http.authorizeRequests()

                .antMatchers("/news/*" ) .authenticated()

                .antMatchers("/*" ) .permitAll()

                 .and()

                 .formLogin()

                .loginPage("/login" )

                .loginProcessingUrl("/login/process" )

                .usernameParameter("username" )

                .passwordParameter("password" )

                .failureUrl("/login?error=true” )

                .and()

                .logout()

                .logoutUrl("/logout" )

                .logoutSuccessUrl("/" )

                .and()

                .csrf()

                .disabled();

    }

    // ...


}

logout() - provides logout support

logoutUrl() - tells Spring what URLs triggers the logout functionality

logoutSuccessUrl() - the URL to which the user will be redirected after a successful logout.

Handling "forbidden" error

Let’s say that we have a page that only anonymous users can view. Commonly, you would want only anonymous users to access the login page. Thus, our configuration will change to something like this:

SecurityConfig.class:

@Configuration

@EnableWebSecurity

@ComponentScan ("com.adtimokhin.security")

public class SecurityConfig extends WebSecurityConfigurerAdapter {


    @Override

    protected void configure (HttpSecurity http) throws Exception {

        http.authorizeRequests()

                .antMatchers("/news/*" ) .authenticated()

                .antMatchers("/login" ) .anonymous() // This is new line of configuration

                .antMatchers("/*" ) .permitAll()

                // And the rest as before ...

    }

    // ...


}

What will we see when an authenticated user tries to enter the “/login” URL? Well, you can check it yourself, but it will probably give you an error 403 - forbidden:

I don’t know about you, but I don’t find this page that interesting for a common user to look at. It is useful in the development stage and all, but we will eventually need to change this to something more user friendly.

One option is to redirect the user to a page that he/she has access to. This can also be configured through the SecurityConfig. It is done like this:

SecurityConfig.class:


@Configuration

@EnableWebSecurity

@ComponentScan ("com.adtimokhin.security")

public class SecurityConfig extends WebSecurityConfigurerAdapter {


    @Override

    protected void configure (HttpSecurity http) throws Exception {

        http.authorizeRequests()

                .antMatchers("/news/*" ) .authenticated()

                .antMatchers("/*" ) .permitAll()

                 .and()

                 .formLogin()

                .loginPage("/login" )

                .loginProcessingUrl("/login/process" )

                .usernameParameter("username" )

                .passwordParameter("password" )

                .failureUrl("/login?error=true” )

                .and()

                .logout()

                .logoutUrl("/logout" )

                .logoutSuccessUrl("/" )

                .and()

                .exceptionHandling() // this is a new line

                .accessDeniedPage("/news/") // this is a new line

                .and()

                .csrf()

                .disabled();

    }

    // ...


}

exceptionHandling() - is a method that lets Spring know that we want to use its integrated excepting handling capabilities.

accessDeniedPage() - method tells to what url the user should be redirected instead of getting shown the 403 page.

Password Encryption

If you go to your user table you will see that anyone who has access to the table can easily read users’ passwords:

"users" table:

Users of your application will not be very happy about it.

We can encrypt the passwords so that no one who can view the table can see passwords.

There is an encoder that we can use - a PasswordEncoder. This is an interface that encodes passwords. We will, therefore, use an implementation of that interface called BCryptPasswordEncoder. We will need to create a bean inside our SecurityConfig:

SecurityConfig.class:

@Configuration

@EnableWebSecurity

@ComponentScan ("com.adtimokhin.security")

public class SecurityConfig extends WebSecurityConfigurerAdapter {


    // ...


    @Bean

    public PasswordEncoder getPasswordEncoder () {

        return new BCryptPasswordEncoder(10) ;

    }


    // ...


}


Strength parameter tells how strong the encoding will be. The bigger the number the harder it will be to crack the password, but it will also take more time to encrypt and decrypt.

I will not go into details of how this encoder encrypts the passwords. It is not an easy topic. But if you are keen to learn more, I cannot stop you. Try to start by reading this Wikipedia article on Bcrypt.

Back to Spring though. Now we have an instrument to encode and decode our passwords. Let’s use it.

We need to encrypt passwords when creating a user. We do that in the implementation of our UserService:

UserServiceImpl.class:

@Component

public class UserServiceImpl implements UserService {


    @Autowired

    private UserRepository userRepository;


    @Autowired

    private PasswordEncoder passwordEncoder;


    @Override

    public void saveUser (User user) {

        String encodedPassword = passwordEncoder.encode(user.getPassword()) ;

        user.setPassword(encodedPassword) ;

        userRepository.save(user) ;

    }



    // ...


}


We need to use our PasswordEncoder to see if the password passed in the login form matches the encoded one. We do that in the authenticate() method inside our AuthProvider. So, let’s update the code for that method as well:

AuthProvider.class:

@Component

public class AuthProvider implements AuthenticationProvider {


    @Autowired

    private UserService userService;


    @Autowired

    private PasswordEncoder passwordEncoder;


    @Override

    public Authentication authenticate (Authentication authentication) throws AuthenticationException {


        String email = authentication.getName() ; // we get the username parameter

        User user = userService.findUser(email) ;


        if (user == null) {

            throw new AuthenticationCredentialsNotFoundException ("User with such email was not detected");

        }


        String password = authentication.getCredentials().toString() ; // we get the credentials that we will turn into string


        if (!passwordEncoder.matches(password, user.getPassword())) {  // this is how we can compare encrypted and not encrypted passwords.

            throw new BadCredentialsException ("Bad credentials");

        }


        ArrayList<GrantedAuthority> authorities = new ArrayList<>() ;

        return new UsernamePasswordAuthenticationToken(user.getEmail(), user.getPassword(), authorities);

    }


    // ...

}



matches() is a method that checks if a raw String matches the encoded one.

CSRF protection

CSRF stands for Cross-Site Request Forgery. It is a type of an online attack. When you publish your web project you will probably have forms in it that will be sent to your application with some user details (POST HTTP requests). Every form triggers a certain types of action to be performed. Some of these forms can only be accessed by authenticated users.

Imagine someone created a website that copies forms from your website.

If the user will be authenticated on your website and activated a form on that malicious website, a certain actiong can be triggered to be performed on your website, though the user didn't intend it (like transferring money to other account, if yours website is a bank).

There is a way to protect our application from such attacks. To each form we send we need to attach a unique key (called token) that will be generated whenever a new form is sent. This token serves as a way to prove that user sends form from your website, because the server will keep track of active tokens. That will mean that after the token has been used it will no longer be valid.

It is very easy to implement such functionality in Spring. Spring will handle the creation and handling of the CSRF tokens. We only need to add the CSRF tokens to all of our forms.

So, let’s do it.

We will start by removing the last three lines from our SecurityConfig that were responsible for blocking CSRF protection.

SecurityConfig.class:


@Configuration

@EnableWebSecurity

@ComponentScan ("com.adtimokhin.security")

public class SecurityConfig extends WebSecurityConfigurerAdapter {


    @Override

    protected void configure (HttpSecurity http) throws Exception {

        http.authorizeRequests()

                .antMatchers("/news/*" ) .authenticated()

                .antMatchers("/*" ) .permitAll()

                 .and()

                 .formLogin()

                .loginPage("/login" )

                .loginProcessingUrl("/login/process" )

                .usernameParameter("username" )

                .passwordParameter("password" )

                .failureUrl("/login?error=true” )

                .and()

                .logout()

                .logoutUrl("/logout" )

                .logoutSuccessUrl("/" )

                .and()

                .exceptionHandling()

                .accessDeniedPage("/news/");

    }

    // ...


}

By default CSRF protection is enabled.

One slight improvement

While testing the system we have so far, I found that after logging in we are directed to the landing page. I want it to change so that we get redirected to “/news/” URL. It is very easy to implement. We need to add a single line in our SecurityConfig:

SecurityConfig.class:


@Configuration

@EnableWebSecurity

@ComponentScan ("com.adtimokhin.security")

public class SecurityConfig extends WebSecurityConfigurerAdapter {


    @Override

    protected void configure (HttpSecurity http) throws Exception {

        http.authorizeRequests()

                .antMatchers("/news/*" ) .authenticated()

                .antMatchers("/*" ) .permitAll()

                 .and()

                 .formLogin()

                .loginPage("/login" )

                .loginProcessingUrl("/login/process" )

                .usernameParameter("username" )

                .passwordParameter("password" )

                .failureUrl("/login?error=true” )

                .defaultSuccessUrl("/news/) // this will redirect us to "/news/" page after successful loging in.

                .and()

                .logout()

                .logoutUrl("/logout" )

                .logoutSuccessUrl("/" )

                .and()

                .exceptionHandling()

                .accessDeniedPage("/news/");

    }

    // ...


}
