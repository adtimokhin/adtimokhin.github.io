Inversion of Control and Dependency Injection

Inversion of Control

Oftenly you will want to use dependencies in your class. Inversion of Control is a programming pattern that helps you to design this type of behaviour in your application effectively.

Let me show you why Inversion of Control (IoC) is useful on an example:

Imagine that you have to write a program for tracking shipment of boxes that contain fragile items.

You will need a Box class where you will also store information about the item inside it. The question is how do you add this fragile item object into your object of Box class?

You have a few options:

Option 1:

Inside Box class create an instance of some fragile item

class Vase {

    // some code ...

}


public class Box {


    Vase vase = new Vase() ;


    public Box () {

        // some code ...

    }


    // some code ...


}

But there is a problem. What if you want to add other fragile object to your Box object? What if you have a second class Mirror? Then you can use a second option

Option 2:

Create an interface or an abstract class that all fragile items will inherit. In this example, we have an interface called Fragile

interface Fragile{}


class Vase implements Fragile {

    // some code ...

}


class Mirror implements Fragile {

    // some code ...

}


public class Box {


    Fragile fragile;


    public Box () {

        fragile = new Vase() ;


        // or


        fragile = new Mirror() ;

    }


    // some code ...


}


Now instead of being limited to a single type of fragile item in our box we can have any we like, unless it implements Fragile interface of course.

But there is still a problem. We create an instance of a class that inherits Fragile inside a Box class. That is not good because we need to update our code all the time when there is a new type of fragile item added to the code.

To solve that we can use option 3.

Option 3:

Pass an object into the instance of Box class as a parameter. We can do it, say, in the constructor or in a setter method.

interface Fragile {

}


class Vase implements Fragile {

    // some code ...

}


class Mirror implements Fragile {

    // some code ...

}


public class Box {


    Fragile fragile;


    public Box (Fragile fragile) {

        this.fragile= fragile;

    }


    public void setFragile (Fragile fragile) {

        this.fragile= fragile;

    }


    // some code ...


}


This option makes our Box class not depenpent on the object we want to put inside it. No matter what object we get in the constructor the code will still make sense. This is what is called the Inversion of Control. You make your Box to have a dependency with an object that gets passed from the outside of the class.

We make a big use of IoC in Spring applications. It allows us to use the same objects in different classes which is very handy.

In Spring we call Java objects that we use in instances of classes “beans”. But there can be only one bean of every class.

We can create beans (and we will) in configuration files and in Java code directly. We will discuss that in greater depth a bit later.

Dependency Injection

Now when we have understood the IoC we should discuss the dependency injection (DI).

I will use the same example with the boxes to explain DI.

In your code you probably would want to create objects of class Box. With the code we have now we will also need to pass a Fragile object into the parameters of the Box object. And it will look something like this:

// ...


class BoxAssembler{

    public void assembleBox () {

        Box box = new Box (new Mirror);

    }

}


But there is a problem. What if we don’t want to create a new Mirror but to use one that exists? What if we need to use the same object in two different classes?

Then we can create a Mirror that we want to use in some container class and pass it from that class every time we need that exact mirror.

class Container{

    public static final Mirror SPECAIL_MIRROR = new Mirror() ;

}


class BoxAssembler{

    public void assembleBox () {

        Box box = new Box(Container.SPECAIL_MIRROR) ;

    }

}


class SpeicalMirrorAssembler{

    public void assembleMirrorBox () {

        Box box = new Box(Container.SPECAIL_MIRROR) ;

    }

}


Now, the problem occurs when you want to add another object to the Container class. What if you want to add a vase, or another fragile object to which you wish to refer in your code?

This is where Spring comes into play. Spring has a special container where it stores all instances of classes that you want to become beans. This helps you to inject the exact dependency you want in any place of your code.

It is most likely that in your Spring application you won’t need 20 objects of the same class to be your beans. Most likely you will need just one instance of a class. But it is still possible to create multiple beans of the same class.

How to work with IoC and DI in Spring

There are a few ways how to create beans:

    • Create beans in configuration files (we will talk about it in the next section)
    • Use annotations.

Annotation is used to mark your code for the program to treat it in some way. Different annotations do different jobs. To create a bean you will need a @Component annotation.

@Component

class Mirror implements Fragile{

    // some code ...

}


Now when the program starts Spring will add an instance of Mirror into its container for us to use

How do you use beans that you created?

Again, there are a few different options, but the most convenient is through @Autowired annotation.

When you add this annotation into a class Spring will automatically inject into your code a bean of that class (if you have a bean of that class, of course).

@Component

class Mirror implements Fragile{

    // some code ...

}


public class Box {

// ...

}


class BoxAssembler{


    @Autowired

    private Mirror mirror;


    public void assmebleBox ()

        Box box = new Box(mirror) ;

    }

}


class SpecialMirrorBoxAssembler{


    @Autowired

    private Mirror mirror;


    public void assembleMirrorBox ()

        Box box = new Box(mirror) ;

    }

}


Now both BoxAssembler and SpecialMirrorBoxAssembler have the instance of the Mirror object.

As I have mentioned, we can work with beans inside configuration files. I will show how to in the next section.
