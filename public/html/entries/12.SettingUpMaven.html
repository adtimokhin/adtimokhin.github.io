Maven

Start working with Maven

In general terms, Maven is a framework which makes the process of compiling a project very simple. Maven is responsible for compiling the code after we press the green start button. We will not get much deeper into what Maven is. For the purpose of this project only opperation with Maven that we will need to do is to add dependencies into a file called pom.xml

pom.xml is a file that contains information needed to build a project. In this file you can specify packaging type, different maven plugins and (more importantly to us) you can add all dependencies you need.

What is a dependency

Dependency allows you to get access in our own program to the code that was written by others. By adding a dependency you will gain access to the code with a functionality you need without a need to write that code by yourself. That is a handy thing. Besides, Spring - is a framework, which means that in order to use it you need to get code for it somewhere.

Organization of pom.xml file

At its core, pom.xml file is a xml file. We will get to know this type of file pretty soon, but for now I will just say that xml files are files that contain a list of properties. Those properties can be used by many parts of a program.

These properties range from something relatively unimportant for the execution of our program (like the name of the developer who has created this project) to the most fundamental properties (like what version of Java we use in our project).

pom.xml stores information for compiling and executing the program.

The format in which that information is stored is simple:

<Name of a property> Data </Name of a property>

Sometimes we put tags inside another tags to symbolise that they are grouped together.

Tags can get more complex, but we will not consider that now.

What is more important, is that now we are able to find where we can store our dependencies.

In the pom.xml file there is a tag <dependencies></dependencies>, and inside it we can put our own dependencies!

Where to find dependencies?

Dependencies are downloaded when you specify all information needed inside the <dependencies></dependencies> tags. We can get that information from this website:

https://mvnrepository.com/

Now, let’s try to download a dependency that we will need in the future.

Step One: Search for a dependency you need

For this example we will download a dependency that will allow us in the future to work with Spring MVC. The dependency is called spring-webmvc

Find the dependency by entering its name:

Click on the first option.

Step Two: copy the dependency into the project

Currently on your screen you have some information about spring-webmvc dependency. There you will see a very big table.

That table contains links to many versions of the code. The newer versions appear at the top of the table. Every version is also accompanied by some information. For example, whether this version is stable (if it is it will not say "alpha" or "beta" next to it) and how many people use this exact version in their projects.

When you follow this tutorial there may be another version of Spring. It is a good idea to take the last stable version of the code.

If you want to understand how release versions work you can read it here. Let's get back to the topic.

Click at the newest stable version (in my case it is 5.3.9).

You will get into a page that looks like this:

There at the bottom where it says “Maven” you will see some xml tags.

Copy those tags.

Now paste them into our pom.xml file between <dependencies></dependencies> tags.

pom.xml

   <!-- ... -->


    <dependencies>


        <!-- This dependency is added automatically if we've used the "archetype approach" -->

        <dependency>

            <groupId>junit</groupId>

            <artifactId>junit</artifactId>

            <version>4.11</version>

            <scope>test</scope>

        </dependency>



        <!-- This is the spring-webmvc dependency that we have copied -->

        <dependency>

            <groupId>org.springframework</groupId>

            <artifactId>spring-webmvc</artifactId>

            <version>5.3.9</version>

        </dependency>


    </dependencies>


   <!-- ... -->

At the bottom right you will be asked by maven if you would like to import changes. Press the “import” option. When you press “import” you are letting your computer download all the files associated with that dependency.

Bravo! You have downloaded your first dependency. All other dependencies are downloaded in a similar manner.

How to understand release versions

Let me also tell you what the version number means. Usually it is made out of 3 numbers. For example:

5.3.9

When programmers update the code they may wish to show that new functions have been added. In that case, the version of the program will change. If a change was small (for example some minor bugs were fixed) then only the last number will increase. If some more significant changes was made (for example some new possibilities were added to the code) then the middle number will increase and the last one will drop to 0. If the change was very big (for example a major release) then the first number will change and all others will drop to 0.

So when there will be a big update in Spring they may decide to publish a version 5.4.0 straight after 5.3.9

If after that Spring developers would want to to publish a major release they might publish it under the number 6.0.0

The latest version of Spring there is when I write these words is Spring 5. It is possible that in future major updates will lead to change in opperation of Spring, and most the information in this tutorial will simply become irrelevant, but probably not all of it. So watch out.

Read this if you want to learn a useful trick for maven

Inside your pom.xml file there are tags <properties></properties> Between them there will be already a few other tags:

pom.xml

   <!-- ... -->


    <properties>

        <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>

        <maven.compiler.source>1.7</maven.compiler.source>

        <maven.compiler.target>1.7</maven.compiler.target>

    </properties>


   <!-- ... -->

You can add your custom tags in between <properties></properties> tags. Those tags that you will add can be used inside the pom.xml file.

Let’s create a tag that we will use to define the version of Spring that we are using (most of the Spring dependencies will be of the same version).

Inside <properties> </properties> tags create a pair of tags :

<spring.version></spring.version>

In between put the version number of your Spring dependencies (for me it is 5.3.9).

pom.xml

   <!-- ... -->


    <properties>

        <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>

        <maven.compiler.source>1.7</maven.compiler.source>

        <maven.compiler.target>1.7</maven.compiler.target>

         <!-- The new bit of code -->

        <spring.version>5.3.9</spring.version>

    </properties>


   <!-- ... -->

Now we can use this property in many of our Spring-related dependencies. To use your own tag write:

${tag.name}

where you need it. Now your Spring dependencies will look something like this:

pom.xml

   <!-- ... -->


        <dependency>

            <groupId>org.springframework</groupId>

            <artifactId>spring-webmvc</artifactId>

            <!-- This is how you may use it in the spring-webmvc dependency -->

            <version>${spring.version}</version>

        </dependency>


   <!-- ... -->

Why do we use archetype to create a project

As promised, we will talk about what exactly was added to the pom.xml file when we had used an archetype to create it.

When you create a project without an archetype it will look different:

File structure:

Your usual pom.xml:

You see that everything looks a bit odd. There is no webapp folder, and pom.xml has much less code in it.

I will not touch on the file structure, but I will talk about the pom.xml.

Project that we have created with help of the archetype maven-archetype-webapp has some important parts to it:

WAR packaging

You see how in our NewsSite project we have <packaging></packaging> tags that have war between them:

pom.xml

   <!-- ... -->

   

    <groupId>com.adtimokhin</groupId>

    <artifactId>newsSite</artifactId>

    <version>1.0-SNAPSHOT</version>

    <packaging>war</packaging>


    <name>newsSite Maven Webapp</name>


   <!-- ... -->

These <packaging></packaging> tags tell how the project should be compiled. That means, how the project’s code will be treated. WAR is one of the ways to compile the code in java. WAR is used for the web applications. Maven will know that you will make a use of technologies that are relevant to the web development when you use WAR packaging. For instance, WEB-INF folder that we have inside the webapp folder is relevant to the web development and is not present in standart java projects. WAR packaging will make sure that WEB-INF folder are treated as a component of your web application. That is the main point for us. Though there are other differences between WAR and JAR (common way for compiling Java code).

Maven plugins

There is a very big part in the archetype-generated pom.xml file that is not present in the other:

pom.xml

   <!-- ... -->


    <build>

        <finalName>newsSite</finalName>

        <pluginManagement><!-- lock down plugins versions to avoid using Maven defaults (may be moved to parent pom) -->

            <plugins>

                <plugin>

                    <artifactId>maven-clean-plugin</artifactId>

                    <version>3.1.0</version>

                </plugin>

                <!-- see http://maven.apache.org/ref/current/maven-core/default-bindings.html#Plugin_bindings_for_war_packaging -->

                <plugin>

                    <artifactId>maven-resources-plugin</artifactId>

                    <version>3.0.2</version>

                </plugin>

                <plugin>

                    <artifactId>maven-compiler-plugin</artifactId>

                    <version>3.8.0</version>

                </plugin>

                <plugin>

                    <artifactId>maven-surefire-plugin</artifactId>

                    <version>2.22.1</version>

                </plugin>

                <plugin>

                    <artifactId>maven-war-plugin</artifactId>

                    <version>3.2.2</version>

                </plugin>

                <plugin>

                    <artifactId>maven-install-plugin</artifactId>

                    <version>2.5.2</version>

                </plugin>

                <plugin>

                    <artifactId>maven-deploy-plugin</artifactId>

                    <version>2.8.2</version>

                </plugin>

            </plugins>

        </pluginManagement>

        <plugins>

            <plugin>

                <groupId>org.apache.maven.plugins</groupId>

                <artifactId>maven-compiler-plugin</artifactId>

                <configuration>

                    <source>8</source>

                    <target>8</target>

                </configuration>

            </plugin>

        </plugins>

    </build>


   <!-- ... -->

This part is used for specifying different plugins that we might want to use for our maven. Maven plugins are what really Maven is used for. Plugins can control the creation of .war files, can be used to compile the code, can be used to create documentation for the file and many other things.

We can decide not to specify Maven plugins like on this example:

Some other pom.xml

However, plugins are still used by Maven (otherwise the project could not run). But when we do specify them we gain a control over what plugins to use and we can even configure them.

So, using the maven archetypes is very handy, and I encourage you to use them when possible so that you don’t mess up your code because you forgot to add properties for .war packaging, for example.
