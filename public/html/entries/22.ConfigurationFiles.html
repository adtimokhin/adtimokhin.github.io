First look at configuration files

Yay! Some actual programming!

Configuration files

Configuration files store information about certain aspects of a program. For example, pom.xml (which is a configuration file) stores information on how to compile the code and what dependencies are used.

We will be writing configuration files that help us to personalise our Spring so it carries the functionality that we want it to carry. In other words it will configure certain aspects of Spring Framework so that it does what we want it to do without writing hundreds lines of code.

As an example, in future in our configuration files we will specify information like what urls we would want to be accessed only by users that have logged into the system. Spring will do the rest to make sure that no unauthenticated user enters the specified urls.

Inside our configuration files we also can create beans that we need. It is a good practise to create beans inside the configuration files when it is easier to create them there. For example, if you need to create a bean of a class that comes inside Spring Framework you can create it inside an appropriate config file.

There are two most commonly used configuration file types that are used in java applications: Java configuration files and XML configuration files.

Let’s explore both options in greater depth! I want to give you a quick introduction to what is there in config files and how it looks. We will not focus right now on the meaning of the code that is inside the files. That is because we will write our own config files and then I will show you the meaning of that code.

Java Configuration files

I want to explain what are the main components of Java configuration by giving you an example from my own code.

Please, do not spend time understanding what this piece of code does. It does not really matter. But what matters are the special annotations and the structre of the Java configuration file itslef.

Some random Java config file:

@Configuration

public class SecurityConfiguration extends WebSecurityConfigurerAdapter {


    @Autowired

    private UserDetailsService userDetailsService;



    @Bean

    public AuthenticationProvider authenticationProvider () {

       // ...

    }


    @Bean

    public BCryptPasswordEncoder bCryptPasswordEncoder () {

        // ...

    }


    @Override

    protected void configure (AuthenticationManagerBuilder auth) throws Exception {

        auth.authenticationProvider(authenticationProvider ()) ;

    }


    @Override

    protected void configure (HttpSecurity http) throws Exception {

        super.configure(http) ;

    }

}


Let’s start with the most essential - all java configuration files are annotated by @Configuration annotation.

Class SecurityConfiguration extends WebSecurityConfigurerAdapter . WebSecurityConfigurerAdapter is a class that is responsible for security configuration in Spring. By extending this class, we can change implementation of some methods, which will eventually personalise how Spring handles security.

And please do not attempt memorizing this. We will cover security aspect in greater depth in the future. Just try to get the understanding how is everything is related.

Usually configuration inside such methods happens by calling different methods on objects that have been passed as arguments.

Like in method configure (AuthenticationManagerBuilder auth) we perform configuration by doing something in that method. That is true for any method inside Java configuration files, especially if they are overriden from other Spring classes. So, in configure (AuthenticationManagerBuilder auth) we call authenticationProvider() method on auth object. That will have an effect on how the program configures. Does not matter what effect exactly.

To create beans inside a java configuration file we simply have to create a method that returns an instance of a class with all parameters we need inside the class annotated with @Configuration. At the top of that method we need to add @Bean so that Spring knows that we declare a bean through that method.

As we have discussed before, you can inject dependencies at any part of your applications by @Autowired annotation. Even inside Java config files.

This is basically it for Java config files. We will write a few of those when we get to writing the actual code.

XML configuration files

Instead of working with classes and methods, all information in xml files is presented as tags.

This is how XML config file may look like:

<?xml version="1.0" encoding="UTF-8"?>

       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"

       xmlns:context="http://www.springframework.org/schema/context"

       xmlns:mvc="http://www.springframework.org/schema/mvc"

       xsi:schemaLocation="http://www.springframework.org/schema/beans http://www.springframework.org/schema/beans/spring-beans.xsd http://www.springframework.org/schema/context http://www.springframework.org/schema/context/spring-context.xsd http://www.springframework.org/schema/mvc http://www.springframework.org/schema/mvc/spring-mvc.xsd">


    <mvc:annotation-driven/>

    <context:component-scan base-package="com.adtimokhin">/>


    <import resource="persistence-config.xml"/>


    <bean id="freemarkerConfig" class="org.springframework.web.servlet.view.freemarker.FreeMarkerConfigurer">

        <property name="templateLoaderPath" value="/WEB-INF/templates"/>

        <property name="defaultEncoding" value="UTF-8"/>

        <property name="freemarkerSettings">

            <props>

                <prop key="default_encoding">UTF-8</prop>

            </props>

        </property>

    </bean>


    <bean id="FreeMarkerViewResolver" class="org.springframework.web.servlet.view.freemarker.FreeMarkerViewResolver">

        <property name="suffix" value=".ftl"/>

        <property name="contentType" value="text/html;charset=UTF-8"/>

        <property name="cache" value="false"/>

    </bean>


</beans>

Most commonly though, you would find Two types of tags:

<bean></bean>tags

Special tags

<bean></bean> tags are used to create beans inside XML configuration files. Here is an example (again, not a meaningful one!):

<bean id="dataSource" class="org.springframework.jdbc.datasource.DriverManagerDataSource">

    <property name="driverClassName" value="${driver}"/>

    <property name="url" value="${url}"/>

    <property name="username" value="${username}"/>

    <property name="password" value="${password}"/>

</bean>

Each tag can have parameters in it. Inside the <bean></bean> tags you would want to add parameters like id (by which the bean can be identified) and class (java class that is used to create that bean). Those parameters can be of almost any type.

Important: You need to id to every bean you create so that you can retrieve it in the future. The reason why we don't specify one in Java configs is because they are created automatically and they will match the Java class name.

So, in essence, parametarization of the bean's contents occurs inside the <bean></bean> tags.

Between <bean></bean> tags you would most usually specify any parameters that bean must have before being created.


Special tags are tags like this one: <tx:annotation-driven/>

They are used not to create beans, but instead are used as indicators for the program to perform different actions.

The sepcial tag above, for example, is used to tell Spring that we want to use annotations inside our program to find classes like the ones with @Component annotation.

But again, you don't have to memroize this.

Specail tags are replaced with special annotations in Java configuration files. You will see an example on a next page.

What to choose Java or XML?

It is totally up to you. You probably would want to get familiar with both types at some point, but I suggest you start with one that is easier for you to understand.

It is believed that Java configuration files are better to use, though.

In my tutorial I will mostly write Java config files. But some will be in XML.

You can even have both Java and XML config files in your code. But usually you stick to one of the two.
