First Controller

And now finally we can begin creating some cool stuff!

Right now let’s do the following. Inside our news website we are going to have many webpages. We will want to have one for some general information about the project (our landing page) and a page where users can create an account for our website.

So I say we create these two pages.

Our landing page would be mapped to “/” url

Our “create account” page will be mapped to “/sign_up” URL.

You don’t have to go for the same URLs, you can choose your own, of course.

Steps to create the pages

Step One: Create controller

A controller is a Java class that calls one of its methods when there is a request to our server. Inside the controller class we specify which method should be called on which URL mapping and HTTP request status (about those we will talk about in just a bit). We also define what that method will do.

Do the following:

    • In the main folder ( com.adtimokhin) Create a folder called controller where we will store all our controllers.
    • Inside that folder create Java class and give it some meaningful name. I will go for AuthController
    • Annotate your class with @Controller annotation, which tells Spring that it is a controller class.

You should end up with something similar:

AuthController.class:

@Controller

public class AuthController {


}


Step Two: create mappings for our two pages

Inside our controller we need to create two methods that will be mapped to two separate URLs, so that we can show two different pages to our users.

Every method that is called when a certain URL is typed should also be annotated. There are a few annotations that you may wish to use. For the purpose of this small project we will consider only two of them:

    • @GetMapping
    • @PostMapping

We use @GetMapping annotation when the action that is going to be carried is targeted to get client some data from server. Like returning a web page or returning some data from our database.

@PostMapping annotation is used when clients adds meaningful information to the server. Like when user gives data to update the context of our database. But note that even when using @PostMapping annotation we still can return our client some data, like a webpage.

Inside the parameters of both @GetMapping and @PostMapping you should pass the actual mapping URL to which the method with such annotation should be called.

If you want to understand this topic better read the following extra-section

Now, let's deicde which annotations to use in our case. When a user enters “/” or “/sign_up” URL we don't expect any meaningful input from users, but we want to give them webpages. Thus, we should be using @GetMapping annotation for both of these methods.

Do the following:

    • Create a method for mapping our landing page.
    • Create a method for mapping our sign up page.

AuthController.class:

@Controller

public class AuthController {


    @GetMapping ("/") // this method will be evoked when user types "/" into the url bar

    public void index () {

        // Method for our landing page.

    }


    @GetMapping ("/sign_up") // this method will be evoked when user types "/sign_up" into the url bar

    public void signUpGet () {

        // Method for our sign up functionality.

    }


}


Of course right now our methods do not return anything useful. So we will not see any landing or sign up pages. In fact, if you try to run the program you will run into an error when you try to go to “/sign_up” mapping.

I will tell you how to fix it in the next section.

Extra sections

Additional section on HTTP Methods

Get and Post are what is called HTTP methods.

HTTP is protocol - a standard way computers communicate with each other.

When you interact with a website, for example, by opening a new page or by filling in some form and pressing the “send” button you actually create an HTTP request.

This request will let the server know that a specific user wants to perform some kind of action with a server.

And these “actions” are marked using HTTP methods.

Here is the list of HTTP methods that are commonly used and what they signal the server to do:

    • GET - get some data back to the user without updating the server’s contents. That can be returning a HTML page
    • POST - add new data on the server. That can be updating contexts of our database, for example.
    • PUT - update existing data on the server.
    • PATCH - update the data stored on the server. Modify only part of the data. Say, change only one parameter (like name or age) stored in DB.
    • DELETE - delete some data from the server

And you can ask yourself, shouldn’t we then be using annotations like:

    • @PutMapping
    • @PatchMapping
    • @DeleteMapping

And the answer is yes, we should, but we will not.

All of the HTTP methods above, apart from GET, can be said to perform the same function as the POST method. By default, HTML supports only GET and POST methods, and that is why we will not get into details of how to send, for example, DELETE requests.

Thus, in our current application we will only be using GET and POST http requests and methods annotated with @GetMapping, with @Postmapping to handle these requests from users.
